# 퍼실리테이션 보드 기획서 (Draft v1)

## 1. 문제 정의

이 프로젝트가 풀려는 문제는 "회의 도구가 부족하다"가 아니라, "회의에서 발언이 골고루 분배되지 않는다"는 것이다. 이 둘은 겉보기에 비슷하지만 해법이 완전히 다르다. 전자라면 기능이 많은 도구(Miro, Mural)가 답이 되지만, 후자라면 답은 오히려 그 반대 — 기능을 줄이고 발언 구조 자체를 바꾸는 쪽에 있다.

실증적으로도 이 문제는 개인의 소극적 성향으로 치부할 수 없다. 직장인의 약 절반이 업무 중 의견을 침묵으로 넘긴다는 조사가 있으며, 그 원인은 두려움(fear)보다 오히려 "말해봐야 바뀌지 않는다"는 무력감(futility)에 가깝다는 분석이 있다(Ethan Burris, UT Austin 연구 인용). 즉 침묵은 회의라는 형식 자체가 구조적으로 만들어내는 결과이지, 개별 참가자의 문제가 아니다. 여기서 이 기획의 전제가 성립한다: 동시에 쓰고 동시에 보는 구조는 "발언 순서를 기다리다 묻히는" 절차적 병목을 원천적으로 없앤다.

따라서 이 프로덕트가 검증해야 할 첫 번째 가설은 다음과 같다.

> 가설 1: 참가자 전원이 동시에 익명 아닌 실명으로, 그러나 순서 없이 동시에 의견을 쓸 수 있게 하면, 발언 편중(소수가 대부분의 발언을 차지하는 현상)이 줄어든다.

## 2. 목표 사용자

- 1차 타깃: 5~15명 규모의 소규모 팀. 정기적으로 회고(retrospective), 브레인스토밍, 우선순위 결정을 진행하는 조직.
- 구체적으로는 프로덕트/디자인 팀, 애자일 코칭 조직처럼 "퍼실리테이션이 업무 프로세스의 일부"인 팀에서 시작하는 것이 타당하다. 이런 팀은 이미 Miro나 FigJam 같은 도구의 무게(가격, 학습 곡선)에 대한 불만을 갖고 있을 가능성이 높기 때문이다.
- 2차 확장 타깃(v1 범위 밖): 사내 워크숍을 비정기적으로 진행하는 HR/기획 부서, 대학 조별 과제.

## 3. 경쟁 환경 분석

| 도구 | 성격 | 강점 | 약점(우리 입장에서의 기회) |
|---|---|---|---|
| Miro / FigJam / Mural | 범용 캔버스 | 기능 최다, 엔터프라이즈 보안·거버넌스 | 무겁고 인당 과금. 회고 하나 돌리기엔 과한 도구. 학습 곡선 존재 |
| EasyRetro / TeleRetro | 회고 전용, 로그인 없이 링크 참여 가능 | 가볍고 무료. 우리 방향과 가장 가까운 경쟁자 | "의견 → 병합 → 문제 정리 → 우선순위 투표"라는 흐름을 강제하지 않고 자유 캔버스 안에서 다 섞어 처리함 |
| Metro Retro | 회고 전용 | 디자인이 캐주얼함 | 무료 플랜 축소 추세(유료 전환 중) |

경쟁 환경에서 확인한 사실 중 이 기획에 중요한 것은 두 가지다.

첫째, 투표 기능에 대한 불만이 FigJam 사용자 커뮤니티에서 반복적으로 제기된다는 점이다. 익명 처리, 인당 투표 수 제한, 자동 집계라는 세 가지가 실사용자들이 원하는 핵심이며, Mural이 이 부분의 사실상 표준으로 언급된다. 우리가 이미 구현한 "항목당 1표, 전체 투표권 제한" 규칙은 표준 닷보팅(중복 허용)과는 다른 선택이라는 점을 이전에 짚었는데, 이 차이를 오히려 포지셔닝으로 가져갈 수 있다 — "소수의 강한 의견이 아니라 팀 전체가 넓게 동의하는 문제를 찾는 도구"라는 메시지.

둘째, 협업 화이트보드 시장 자체는 조사 기관마다 추정치 편차가 크지만(2025~2026년 기준 시장 규모를 20억~45억 달러 사이로, 연평균 성장률을 12~20%대로 각각 추정), 공통적으로 원격/하이브리드 근무 확산이 성장 동력이라는 점은 일치한다. 다만 이 숫자는 "범용 화이트보드 시장" 전체이며, "회고/퍼실리테이션 전용 소도구" 시장의 크기는 별도로 확인된 자료가 없다 — 이는 추가 조사가 필요한 부분으로 7장에 남겨둔다.

## 4. 차별점 (가치 제안 가설)

경쟁 도구 대비 이 프로덕트가 주장할 수 있는 차별점은 다음 네 가지다. 다만 이것들은 아직 "우리가 만든 것"일 뿐 "시장이 원하는 것"이라는 증거는 없다 — 6장의 검증 항목으로 다시 다룬다.

1. **로그인 없음, 이름만으로 참여**: 온보딩 마찰을 없애 소집 즉시 시작 가능
2. **범용 캔버스가 아니라 하나의 흐름에 특화**: 의견 → 병합 → 문제 정리 → 우선순위 투표라는 순서를 강제해, 캔버스가 산으로 가는 것을 막음
3. **프로젝트별 저장/재방문**: 회고는 반복되는 행위인데, 매번 새로 만드는 대신 팀의 이력이 쌓이는 구조
4. **결과를 이미지 한 장으로 요약**: 회의 후 공유·아카이빙을 위해 별도 정리 작업이 필요 없음

## 5. MVP 범위

| 구분 | 항목 | 비고 |
|---|---|---|
| Must (v1에 반드시 있어야) | 이름 기반 참여, 색상 자동 배정 | 이미 구현 |
| Must | 포스트잇 작성/삭제, 같은 주제 안에서 병합 | 이미 구현 |
| Must | 문제 정리 리스트 | 이미 구현 |
| Must | 우선순위 투표(항목당 1표, 전체 투표권 제한) | 이미 구현. 표준 닷보팅과 다른 규칙임을 온보딩 문구로 설명 필요 |
| Must | 프로젝트별 저장/재방문 | 이미 구현 |
| Should (v1에 있으면 좋음, 없어도 출시 가능) | 진행 과정 요약 이미지 내보내기 | 이미 구현 |
| Should | 다중 의견 보드(주제별로 나눠 받기) | 이미 구현. 다만 실제로 몇 팀이 "주제를 나눠서" 받고 싶어 하는지는 검증 안 됨 |
| Could (v2 이후) | 실시간 동기화 고도화(폴링 → 웹소켓) | 인원이 많아질수록(10명 이상 동시 편집) 폴링 방식의 한계가 드러날 가능성 |
| Could | 익명 모드 | 경쟁사 요청이 많은 기능이나, "이름 기반"이 우리 차별점과 충돌할 수 있어 신중히 검토 |
| Out of scope (v1에서 명시적으로 제외) | 화상회의 연동, Jira/Slack 연동, 템플릿 라이브러리 | 범용 캔버스와 경쟁하는 방향이 아니므로 제외 |

## 6. 검증해야 할 가설과 지표

이 기획서의 약점은 "우리가 만들고 싶은 것"과 "팀이 실제로 필요로 하는 것"을 아직 구분하지 못했다는 점이다. 다음 가설들은 실제 사용 데이터 없이는 참인지 거짓인지 알 수 없으며, 그 자체가 검증 대상이다.

- 가설 A: 로그인 없는 참여가 실제 세션 시작률(초대 링크 클릭 → 이름 입력 완료)을 높인다.
- 가설 B: "항목당 1표" 규칙이 팀 만족도를 낮추지 않는다 (표준 닷보팅에 익숙한 사용자가 불편해할 가능성 존재).
- 가설 C: 프로젝트 재방문(같은 팀이 두 번째 세션을 같은 프로젝트 안에서 진행)이 실제로 발생한다.

측정 가능한 지표(초기 MVP 배포 후 수집 대상):
- 세션 완료율: 참여 시작 대비 투표 단계 도달 비율
- 세션당 평균 참여자 수, 세션당 평균 포스트잇 수
- 재방문율: 프로젝트 목록에서 기존 프로젝트를 다시 여는 비율
- 이미지 내보내기 사용률: 이 기능이 실제로 쓰이는지, 아니면 "있으면 좋지만 안 씀"인지 구분

## 7. 리스크 및 추가로 조사가 필요한 사항

- **동시 편집 한계**: 현재는 2초 폴링 기반 동기화다. 10명 이상이 동시에 포스트잇을 쓰거나 투표할 때 지연·충돌이 체감될 수 있다. 실사용 규모(팀당 인원수)를 먼저 확정해야 이 부분의 우선순위를 정할 수 있다.
- **회고 전용 소도구 시장의 실제 크기 불명**: 3장에서 언급했듯, 범용 화이트보드 시장 수치는 있으나 "가벼운 퍼실리테이션 전용 도구"만의 시장 규모·성장률 자료는 찾지 못했다. EasyRetro, TeleRetro 등 개별 기업의 사용자 수·매출 공개 자료도 제한적이다 — 이 부분은 정성적 인터뷰(잠재 사용자 5~10팀)로 보완하는 편이 시장 리포트를 더 찾는 것보다 효율적일 수 있다.
- **차별점의 지속가능성**: "로그인 없음"과 "단일 워크플로우 특화"는 Miro/FigJam이 마음만 먹으면 쉽게 따라할 수 있는 종류의 차별점이다. 진짜 방어 가능한 차별점은 기능이 아니라 "이 도구를 쓰면 팀의 퍼실리테이션 습관 자체가 어떻게 바뀌는가"에 있을 가능성이 크며, 이는 지금 단계에서는 가설로만 존재한다.
