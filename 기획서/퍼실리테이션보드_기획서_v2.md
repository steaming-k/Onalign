# 퍼실리테이션 보드 기획서 (Draft v2 — 아이데이션·회고 연결)

## 1. 문제 정의

이 프로젝트가 풀려는 문제는 "회의 도구가 부족하다"가 아니라, "회의에서 발언이 골고루 분배되지 않는다"는 것이다. 이 둘은 겉보기에 비슷하지만 해법이 완전히 다르다. 전자라면 기능이 많은 도구(Miro, Mural)가 답이 되지만, 후자라면 답은 오히려 그 반대 — 기능을 줄이고 발언 구조 자체를 바꾸는 쪽에 있다.

실증적으로도 이 문제는 개인의 소극적 성향으로 치부할 수 없다. 직장인의 약 절반이 업무 중 의견을 침묵으로 넘긴다는 조사가 있으며, 그 원인은 두려움(fear)보다 오히려 "말해봐야 바뀌지 않는다"는 무력감(futility)에 가깝다는 분석이 있다(Ethan Burris, UT Austin 연구 인용). 즉 침묵은 회의라는 형식 자체가 구조적으로 만들어내는 결과이지, 개별 참가자의 문제가 아니다. 여기서 이 기획의 전제가 성립한다: 동시에 쓰고 동시에 보는 구조는 "발언 순서를 기다리다 묻히는" 절차적 병목을 원천적으로 없앤다.

따라서 이 프로덕트가 검증해야 할 첫 번째 가설은 다음과 같다.

> 가설 1: 참가자 전원이 동시에 익명 아닌 실명으로, 그러나 순서 없이 동시에 의견을 쓸 수 있게 하면, 발언 편중(소수가 대부분의 발언을 차지하는 현상)이 줄어든다.

## 2. 목표 사용자

- 1차 타깃: 5~15명 규모의 소규모 팀. 정기적으로 회고(retrospective), 브레인스토밍, 우선순위 결정을 진행하는 조직.
- 구체적으로는 프로덕트/디자인 팀, 애자일 코칭 조직처럼 "퍼실리테이션이 업무 프로세스의 일부"인 팀에서 시작하는 것이 타당하다. 이런 팀은 이미 Miro나 FigJam 같은 도구의 무게(가격, 학습 곡선)에 대한 불만을 갖고 있을 가능성이 높기 때문이다.
- 2차 확장 타깃(v1 범위 밖): 사내 워크숍을 비정기적으로 진행하는 HR/기획 부서, 대학 조별 과제.

## 3. 경쟁 환경 분석

메인 경쟁자를 EasyRetro와 TeleRetro로 재설정한다. Miro/FigJam/Mural은 참고 대상이지 직접 경쟁자로 보지 않는다 — 가격대와 학습 곡선이 다른 체급이기 때문이다.

| 도구 | 성격 | 강점 | 약점(우리 입장에서의 기회) |
|---|---|---|---|
| EasyRetro | 회고 전용, 로그인 없이 링크 참여 | 가볍고 무료, 250개 이상 템플릿 | 회고 세션 하나하나가 독립적. 이전 세션(특히 그 팀의 킥오프·아이데이션)과 연결되지 않음 |
| TeleRetro | 회고 전용, 링크 참여, 펄스 서베이로 세션 사이 팀 상태 추적 | 회고 사이 텀을 메우는 기능(Pulse Survey)을 갖춘 몇 안 되는 도구 | 세션 간 연결은 "팀 분위기 추적"에 한정. 특정 프로젝트의 목표·우선순위와는 연결되지 않음 |
| ClickUp / monday.com / Notion (참고) | 범용 업무관리 + 회고 템플릿 | 회고 결과를 실제 작업(Task)과 연결 가능 | 회고 전용 도구가 아니라서 무겁고, 워크숍 특유의 동시-작성 경험이 약함 |

이번 리서치에서 확인한 중요한 사실 하나: **"아이데이션(프로젝트 시작 시점의 우선순위 결정)"과 "회고(프로젝트 종료 시점의 회고)"를 같은 프로젝트 안에서 가볍게 이어주는 도구는 조사 범위 안에서 발견되지 않았다.** 회고 전용 도구(EasyRetro, TeleRetro, Neatro, GoRetro)는 회고만 다루고, 생애주기 전체를 다루는 도구(ClickUp, monday.com, Notion)는 회고를 "여러 기능 중 하나"로만 취급해 로그인 없는 가벼운 워크숍 경험을 제공하지 않는다. 이 빈틈이 5장에서 다룰 새 방향의 근거다.

기존에 확인했던 사실들도 유효하다: 투표 기능(익명 처리·인당 투표수 제한·자동 집계)에 대한 불만이 FigJam 커뮤니티에서 반복 제기되며, 협업 화이트보드 시장 자체는 조사 기관마다 편차는 있으나(2025~2026년 기준 20억~45억 달러, 연 12~20%대 성장) 원격/하이브리드 근무 확산이 공통 성장 동력으로 지목된다. 다만 이 숫자는 범용 화이트보드 시장 전체이며, 회고 전용 소도구 시장만의 크기는 별도 자료가 없다.

## 4. 차별점 (가치 제안 가설)

경쟁 도구 대비 이 프로덕트가 주장할 수 있는 차별점은 다음 다섯 가지다. 다만 이것들은 아직 "우리가 만든 것"일 뿐 "시장이 원하는 것"이라는 증거는 없다 — 6장의 검증 항목으로 다시 다룬다.

1. **로그인 없음, 이름만으로 참여**: 온보딩 마찰을 없애 소집 즉시 시작 가능
2. **범용 캔버스가 아니라 하나의 흐름에 특화**: 의견 → 병합 → 문제 정리 → 우선순위 투표라는 순서를 강제해, 캔버스가 산으로 가는 것을 막음
3. **프로젝트별 저장/재방문**: 회고는 반복되는 행위인데, 매번 새로 만드는 대신 팀의 이력이 쌓이는 구조
4. **결과를 이미지 한 장으로 요약**: 회의 후 공유·아카이빙을 위해 별도 정리 작업이 필요 없음
5. **(신규) 아이데이션과 회고의 연결**: 프로젝트 시작 시점에 우선순위 투표로 뽑은 문제를, 프로젝트 종료 시점의 회고에서 다시 불러와 "그때 그 문제, 실제로 해결됐나?"를 묻는다. 3장에서 확인했듯 이 연결을 가볍게 제공하는 도구는 조사 범위 안에서 없었다 — 다섯 가지 중 가장 방어력이 큰 차별점일 가능성이 있다.

## 5. 신규 기능 설계: 아이데이션 → 회고 연결

이번에 제안하신 "프로젝트 완료 버튼 → 회고 작성" 아이디어를 구체적인 흐름으로 정리하면 다음과 같다.

1. 프로젝트는 `진행 중` / `완료` 상태를 갖는다. 지금까지 만든 화면(의견 작성 → 문제 정리 → 우선순위 투표)은 모두 `진행 중` 상태에서 이뤄지는 아이데이션 단계다.
2. 프로젝트 화면에 "프로젝트 완료" 버튼을 추가한다. 이 버튼을 누르면 프로젝트 상태가 `완료`로 바뀌고, 새로운 `회고` 탭이 열린다.
3. 회고 탭은 KPT(Keep-Problem-Try) 같은 표준 회고 포맷으로 시작한다 — 처음부터 새 포맷을 설계하기보다 이미 팀들이 익숙한 형식을 쓰는 편이 학습 비용이 낮다.
4. **핵심 차별화 지점**: 회고 탭 상단에 아이데이션 단계에서 우선순위 투표로 뽑힌 상위 문제 목록을 자동으로 보여준다. 참가자는 그 목록을 보면서 "해결됨 / 부분 해결 / 미해결"을 체크하고, 그 판단을 근거로 KPT를 작성한다. 즉 회고가 빈 캔버스에서 시작하지 않고, 자신들이 프로젝트 시작 시점에 스스로 정한 기준에서 시작한다.
5. 완료 후에는 기존 "이미지로 저장" 기능을 확장해, 아이데이션 요약과 회고 요약을 한 장의 타임라인 이미지로 함께 내보낸다 — "우리가 뭘 하기로 했고, 실제로 뭐가 됐는가"를 한 장으로 보여주는 것이 최종 목표다.

이 설계에서 의도적으로 뺀 것: 액션 아이템 담당자 지정, 마감일, Jira 연동 같은 "실행 추적" 기능이다. 이건 ClickUp·monday.com이 이미 잘하는 영역이고, 이 도구가 그쪽으로 확장하는 순간 범용 업무관리 도구와 정면으로 경쟁하게 된다 — 그건 이 도구의 체급을 벗어나는 일이다.

## 6. MVP 범위

먼저 짚어야 할 것: 지난번 우리는 MVP를 좁히는 방향으로 다듬고 있었는데, 이번 요청은 반대로 범위를 넓히는 방향이다. 이 자체가 나쁜 신호는 아니다 — "좁고 정확한 아이데이션 도구"에서 "아이데이션+회고를 잇는 도구"로 포지셔닝이 바뀌면, 회고 기능은 부가 기능이 아니라 핵심 경쟁력이 되기 때문이다. 다만 두 가지를 한 번에 다 만들면 검증 주기가 늘어지므로, 아래처럼 단계를 쪼갰다.

| 구분 | 항목 | 비고 |
|---|---|---|
| Must (v1) | 이름 기반 참여, 색상 자동 배정 | 이미 구현 |
| Must | 포스트잇 작성/삭제, 같은 주제 안에서 병합 | 이미 구현 |
| Must | 문제 정리 리스트, 우선순위 투표 | 이미 구현 |
| Must | 프로젝트별 저장/재방문, 프로젝트 상태(진행 중/완료) | 상태 필드만 추가하면 됨 |
| Must | "프로젝트 완료" 버튼 → 단순 KPT 회고 탭 오픈 | 신규. 표준 포맷을 쓰는 대신 만드는 시간을 아낌 |
| Must | 회고 탭에 아이데이션 우선순위 문제 목록 자동 표시 + 해결 여부 체크 | 신규. 이게 없으면 "그냥 회고 템플릿 하나 얹은 것"과 다르지 않아, 차별점이 사라짐 |
| Should | 진행 과정 요약 이미지 내보내기(아이데이션+회고 통합) | 기존 기능 확장 |
| Should | 다중 의견 보드(주제별로 나눠 받기) | 실제 수요 검증 안 됨 |
| Could (v2 이후) | 실시간 동기화 고도화(폴링 → 웹소켓) | 인원 규모 늘어날 때 재검토 |
| Could | 회고 포맷 다양화(4Ls, Sailboat 등 선택 가능하게) | 처음엔 KPT 하나로 충분, 수요 보고 확장 |
| Could | 익명 모드 | "이름 기반"이라는 차별점과 충돌 가능, 신중 검토 |
| Out of scope | 액션 아이템 담당자·마감일 지정, Jira/Slack 연동, 화상회의 연동 | 5장에서 설명한 대로 범용 업무관리 도구 영역이라 의도적으로 제외 |

## 7. 검증해야 할 가설과 지표

이 기획서의 약점은 "우리가 만들고 싶은 것"과 "팀이 실제로 필요로 하는 것"을 아직 구분하지 못했다는 점이다. 다음 가설들은 실제 사용 데이터 없이는 참인지 거짓인지 알 수 없으며, 그 자체가 검증 대상이다.

- 가설 A: 로그인 없는 참여가 실제 세션 시작률(초대 링크 클릭 → 이름 입력 완료)을 높인다.
- 가설 B: "항목당 1표" 규칙이 팀 만족도를 낮추지 않는다 (표준 닷보팅에 익숙한 사용자가 불편해할 가능성 존재).
- 가설 C: 프로젝트 재방문(같은 팀이 두 번째 세션을 같은 프로젝트 안에서 진행)이 실제로 발생한다.
- 가설 D (신규): 같은 팀이 아이데이션에 쓴 도구를 몇 주/몇 달 뒤 회고에도 그대로 쓴다 — 즉 "프로젝트 완료" 버튼까지 도달하는 팀이 실제로 존재한다. 이건 가장 위험한 가설이다. 아이데이션과 회고 사이 간격이 길면(수 주~수 개월), 팀이 애초에 이 도구로 돌아온다는 보장이 없다.

측정 가능한 지표(초기 MVP 배포 후 수집 대상):
- 세션 완료율: 참여 시작 대비 투표 단계 도달 비율
- 세션당 평균 참여자 수, 세션당 평균 포스트잇 수
- 재방문율: 프로젝트 목록에서 기존 프로젝트를 다시 여는 비율
- 이미지 내보내기 사용률: 이 기능이 실제로 쓰이는지, 아니면 "있으면 좋지만 안 씀"인지 구분
- **프로젝트 완료 전환율(신규)**: 아이데이션을 시작한 프로젝트 중 실제로 "완료" 버튼을 눌러 회고까지 도달하는 비율. 이 숫자가 낮다면, 두 기능을 하나의 프로젝트로 묶은 설계 자체가 틀렸다는 신호다.

## 8. 리스크 및 추가로 조사가 필요한 사항

- **동시 편집 한계**: 현재는 2초 폴링 기반 동기화다. 10명 이상이 동시에 포스트잇을 쓰거나 투표할 때 지연·충돌이 체감될 수 있다. 실사용 규모(팀당 인원수)를 먼저 확정해야 이 부분의 우선순위를 정할 수 있다.
- **회고 전용 소도구 시장의 실제 크기 불명**: 범용 화이트보드 시장 수치는 있으나 "가벼운 퍼실리테이션 전용 도구"만의 시장 규모·성장률 자료는 찾지 못했다. EasyRetro, TeleRetro 등 개별 기업의 사용자 수·매출 공개 자료도 제한적이다 — 이 부분은 정성적 인터뷰(잠재 사용자 5~10팀)로 보완하는 편이 시장 리포트를 더 찾는 것보다 효율적일 수 있다.
- **차별점의 지속가능성**: "로그인 없음"과 "단일 워크플로우 특화"는 Miro/FigJam이 마음만 먹으면 쉽게 따라할 수 있는 종류의 차별점이다. 아이데이션-회고 연결(4장의 5번)이 상대적으로 방어력이 크지만, 이 역시 EasyRetro나 TeleRetro가 "이전 세션 불러오기" 기능 하나만 추가해도 좁혀질 수 있는 격차다.
- **범위 확장에 따른 리스크(신규)**: 아이데이션 도구와 회고 도구를 하나로 묶는 것은 "한 곳에서 다 된다"는 장점과 동시에 "둘 다 어중간해진다"는 위험을 같이 가진다. EasyRetro/TeleRetro 사용자가 굳이 이 도구로 옮겨올 이유는 "회고가 낫다"가 아니라 "아이데이션부터 회고까지 잇는 게 이 도구에서만 된다"여야 한다 — 회고 기능 자체의 완성도만으로는 두 전문 도구를 이기기 어렵다는 뜻이다.
- **아이데이션-회고 간격 문제**: 두 시점 사이 간격이 길어질수록(예: 분기 단위 프로젝트) 팀이 같은 프로젝트로 돌아올 확률이 낮아질 수 있다. 반대로 스프린트 단위(1~2주)처럼 간격이 짧은 팀에서는 이 연결이 훨씬 자연스러울 수 있다 — 2장의 목표 사용자를 "스프린트 단위로 일하는 팀"으로 더 좁혀야 할 수도 있다.
